### 2.2 Надёжность

| Проблема | Где | Критичность | Описание |
|----------|-----|-------------|----------|
| **Нет бэкапов БД** | Нигде | 🔴 Высокая | `data/protocol.db` — единственный источник. Нет cron-задачи для бэкапа, нет репликации |
| **Нет ротации логов** | `bot.log` | 🟡 Средняя | `logging.basicConfig` пишет в StreamHandler → systemd journal. Файл `bot.log` растет бесконечно |
| **FTS5 триггеры могут упасть** | `database.py` SCHEMA | 🟡 Средняя | Триггеры `fts_insert` и `fts_delete` не имеют обработки ошибок. Если FTS5 сломается, синхронизация нарушится |
| **Нет офлайн-режима** | `classifier.py` | 🟡 Средняя | Если OpenRouter недоступен, создание фрагментов падает. `_fallback_result` возвращает `unknown_type: true`, но фрагмент всё равно сохраняется с пустой классификацией |
| **Один воркер uvicorn** | `protocol.service` | 🟢 Низкая | `ExecStart=/usr/local/bin/uvicorn main:app --host 0.0.0.0 --port 8000` — нет `--workers`. Для 1 пользователя хватит |

### 3.1 Что работает но неправильно

| Проблема | Где | Описание |
|----------|-----|----------|
| **Два `get_db`** | `app/database.py` и `app/deps.py` | В `database.py` — `async def get_db()` (возвращает соединение). В `deps.py` — `async def get_db()` (контекстный менеджер). FastAPI зависимости используют версию из `deps.py`, но `database.py` тоже имеет свою. Путаница |
| **`get_user_by_tg_id` возвращает только id** | `database.py` строка 129 | Функция возвращает `{"id": row["id"]}`, хотя в `deps.py` ожидают полный объект с `tg_id`, `plan`, `meta`. Бот вызывает `get_user_by_tg_id`, но если понадобятся другие поля — их нет |
| **Магический триггер считает глобально** | `fragments.py` строка 99 | `SELECT COUNT(*) FROM memory_fragments WHERE user_id=?` — считает правильно по user_id, НО `magic_trigger` передается в ответе как флаг. Если пользователей станет много, логика верна, но сейчас это неочевидно из кода |
| **Нет закрытия httpx клиента в боте при ошибке** | `bot/main.py` | `http_client` создается глобально, `on_shutdown` закрывает его. Но при необработанном исключении может не закрыться |
| **Конвертация аудио блокирует event loop** | `classifier.py` строка 86 | `subprocess.run(["ffmpeg"...])` — синхронный вызов в async коде. Может блокировать при нагрузке |

### 3.2 Что надо переписать до масштабирования

| Компонент | Проблема | Что делать |
|-----------|----------|------------|
| **SQLite → PostgreSQL** | SQLite не держит конкурентные записи от многих пользователей | Менять `database.py` на asyncpg или SQLAlchemy async. FTS5 заменить на полнотекстовый поиск в PostgreSQL или отдельный ElasticSearch |
| **Классификатор** | Жестко завязан на OpenRouter, промпты в том же файле | Вынести промпты в конфиг/БД, сделать абстракцию LLM-провайдера (интерфейс с методом `classify`) |
| **Ночной аналитик** | Обрабатывает пользователей последовательно, один HTTP-клиент | Сделать конкурентную обработку (asyncio.gather), пул HTTP-клиентов |
| **Отсутствие пула соединений БД** | Каждый запрос = новое соединение SQLite | Для SQLite не актуально (один файл), но при переезде на PostgreSQL нужен пул (SQLAlchemy pool) |
| **Нет кэширования** | Каждый запрос к БД, классификация не кэшируется | Добавить Redis для кэша паттернов, результатов классификации дубликатов |

### 3.3 Что можно не трогать пока

| Компонент | Почему |
|-----------|--------|
| **Бэкап-файлы бота** (`main.py.backup*`) | Это история версий, не мешают |
| **Скрипты обслуживания** (`check_db.py`, `cleanup.sh` и др.) | Полезны для отладки, не влияют на продакшн |
| **Архив `protocol.zip`** | Старые версии, можно удалить, но и так ОК |

| **`__pycache__`** | Стандартный кэш Python, добавить в `.gitignore` |

## 5. АРХИТЕКТУРНЫЕ РИСКИ

### 5.1 Где SQLite станет узким местом

1. **Конкурентные записи**: SQLite с WAL может обрабатывать много чтений и 1 запись одновременно. При 5+ пользователях, пишущих одновременно, будут блокировки.
   - **Признак**: появление `sqlite3.OperationalError: database is locked`
   - **Решение**: переезд на PostgreSQL

2. **Размер БД**: SQLite работает до 1TB, но FTS5 начнет тормозить на >100k фрагментов.
   - **Признак**: запросы `/search` становятся медленнее 500ms
   - **Решение**: индексы FTS5 ребилд, или переезд на ElasticSearch

3. **Отсутствие connection pooling**: каждый запрос FastAPI открывает новое соединение.
   - **Признак**: замедление при >50 RPS
   - **Решение**: для SQLite не критично (локальный файл), для PostgreSQL нужен пул

### 5.2 Где текущий стек не выдержит роста

| Сценарий роста | Лимит текущего стека | Что упадет первым |
|----------------|---------------------|-------------------|
| **Пользователи > 100** | Один процесс uvicorn, SQLite | SQLite заблокируется на записи |
| **Запросов > 50/сек** | Один воркер FastAPI | API начнет таймиться (default timeout 60s) |
| **Фрагментов > 10k на юзера** | FTS5 в SQLite, нет пагинации в `/today` | Ночной аналитик начнет жрать память (все фрагменты за 24ч в память) |
| **Голосовых > 100/день** | ffmpeg синхронно, OpenRouter rate limits | Бот начнет таймиться на транскрибации |

### 5.3 Что надо заложить сейчас чтобы не переписывать потом

1. **Абстракция БД (Repository pattern)**
   - Сейчас SQL-запросы разбросаны по `database.py`, `routers/*.py`
   - Надо: создать классы `FragmentRepository`, `UserRepository` с методами `save()`, `find_by_id()`
   - Это позволит потом просто сменить драйвер БД

2. **Конфигурация окружений**
   - Сейчас `.env` один, `config.py` читает переменные плоско
   - Надо: `pydantic-settings` уже есть, использовать `env_file = [".env", ".env.prod"]`, добавить префиксы `PROD_`, `DEV_`

3. **Структурированное логирование**
   - Сейчас `logging.basicConfig`, формат строкой
   - Надо: `structlog` или `json-log-formatter`, чтобы потом парсить логи в ELK/Grafana

4. **Метрики и здоровье**
   - Сейчас `/health` возвращает `{"status": "ok"}`
   - Надо: добавить проверку БД (`SELECT 1`), проверку OpenRouter (тестовый запрос), метрики в `/metrics` (prometheus format)

## ИТОГОВЫЙ ВЕРДИКТ

**Общая оценка**: 🟢 **Хорошо для MVP / личного использования**

Проект функционален, архитектура понятна, код читаемый. Для использования ЗавЛабом одному — более чем достаточно.

**Главные риски**:
1. Отсутствие бэкапов (риск потери всех данных)
2. Слабая авторизация (если бот попадет в публичный доступ)
3. SQLite не масштабируется (но для 1-2 юзеров — ОК)

**Приоритет действий**:
1. Сделать бэкап БД (прямо сейчас)
2. Сменить SECRET_KEY
3. Добавить JWT
4. Активировать фичи из WIP
5. Думать о масштабировании только при росте пользователей >10
