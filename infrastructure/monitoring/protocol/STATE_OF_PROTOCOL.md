## 7. РИСКИ

### Что может сломаться из-за моих изменений:

1. **БАГ в night_analyst.py строка 144:**
   - `new_patterns` содержит строки (desc), а не словари
   - `p[:80]` сработает, но `p` — это строка, а не объект с полями
   - **Исправление:** заменить `new_patterns.append(desc)` на `new_patterns.append(pattern)` и форматировать словарь

2. **HTTP-запросы в night_analyst.py:**
   - Если `BOT_TOKEN` не задан → сводки не улетят (есть проверка, но логи могут засориться)
   - Если Telegram API недоступен → упадёт с ошибкой (есть try/except, но успокаиваться не стоит)

3. **Онбординг делает 2 запроса к API:**
   - `POST /fragments` + `GET /fragments?count_only=true`
   - При нагрузке может тормозить (но пока фрагментов мало — норм)

4. **Таймер сработает через ~30 минут:**
   - Если сводка упадёт → юзеры не получат уведомление
   - Нужно проверить логи после 09:00 MSK: `journalctl -u protocol-analyst -f`

5. **SQL в `get_user_by_id`:**
   - `"SELECT id, tg_id, plan FROM users WHERE id = ?"` — проверить, что поле `tg_id` существует в БД
   - (В схеме есть: `tg_id TEXT UNIQUE`)

6. **Frontend People.tsx может быть пустым:**
   - Если `peopleAPI.listPeople()` вернёт пустой список → покажет "Пока нет людей"
   - Нужно добавить людей через фрагменты (LLM должен извлекать имена)
